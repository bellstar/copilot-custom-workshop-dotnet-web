# Token 節約とコンテキスト管理

[前へ - MVC 実装 + Vision](../6_ImplementMVC/README_JA.md) | [次へ - ユニットテスト](../8_UnitTesting/README_JA.md)

Copilot が正確な出力を返すかどうかは、**どのようなコンテキスト（文脈情報）を渡すか** に大きく左右されます。このステップでは、Step 5・6 で実装したコードを題材に、効率的にコンテキストを管理し、Token を節約しながら高品質な出力を得る方法を学びます。

> **振り返りのステップ:** Step 5・6 では Agent モードで一気に実装しました。このステップでは一歩引いて、「より良いプロンプトの書き方」を実験で体感します。

---

## なぜコンテキスト管理が重要なのか

| 問題 | 原因 | 結果 |
|------|------|------|
| 出力が的外れ | 必要な情報が不足 | 既存コードと矛盾する提案 → 手戻り発生 |
| 応答が遅い | 不要な情報を大量に送信 | 検索・推論に時間がかかる |
| 手戻りの連鎖 | 曖昧な指示 | 修正プロンプトを何度も投げ、累積コストが増大 |
| 途中で切れる | コンテキストウィンドウの上限超過 | 不完全なコード生成 |

**目標:** 必要十分な情報だけを渡し、**1 回で正確な出力**を得る。これが結果的に時間もクレジットも節約する。

---

## 比較実験：コンテキストの渡し方で出力がどう変わるか

以下の 3 パターンで同じ質問をして、違いを体感してください。

> **❗ この実験は Ask モードで行います。** Agent モードではファイルが変更されてしまうため、出力を観察・比較する目的には Ask モードが適切です。

> **❗ パターンごとに必ず新しいチャットセッションを開いてください。** 同じセッション内で続けて質問すると、前の回答がコンテキストとして残り、純粋な比較ができません。新しいセッションは Chat パネル上部の「＋」ボタンで開けます。

### 実験用プロンプト

「`AppDbContext` に猫の初期データ 5 件をシード投入する拡張メソッドを作成して」

この質問は、Step 5 で作成した `AppDbContext` と `Cat` エンティティの実装を正しく参照できるかどうかでコードの正確さが大きく変わります。

---

### パターン A: コンテキストなし

**新しいチャットセッション**を開き、以下を入力してください：

> 💬 **Ask モード**

```
AppDbContext に猫の初期データ 5 件をシード投入する拡張メソッドを作成して
```

**予想される問題:**
- `AppDbContext` や `Cat` のプロパティ構成がわからないので推測で書く
- namespace やクラス名が不正確になる
- `Cat` に存在しないプロパティ（例: `Color`）を使う可能性がある

---

### パターン B: `#codebase` で全体を渡す

**新しいチャットセッション**を開き、以下を入力してください：

> 💬 **Ask モード**

```
#codebase AppDbContext に猫の初期データ 5 件をシード投入する拡張メソッドを作成して
```

**特徴:**
- Copilot がワークスペース全体を検索して `AppDbContext` と `Cat` を見つける
- 正確な出力が得られやすいが、検索・処理に時間がかかる
- 大規模プロジェクトでは Token 消費が大きい

---

### パターン C: `#file` でピンポイント指定

**新しいチャットセッション**を開き、以下を入力してください：

> 💬 **Ask モード**

```
#file:MeowWorld/Data/AppDbContext.cs と #file:MeowWorld/Models/Cat.cs を参照して、
AppDbContext に猫の初期データ 5 件をシード投入する拡張メソッドを作成して
```

**特徴:**
- 必要なファイルだけを明示的に渡す
- `Cat` の実際のプロパティ定義を参照するので最も正確
- Token 消費が最小限

---

### 比較結果

| 観点 | A: なし | B: #codebase | C: #file |
|------|---------|-------------|----------|
| 出力の正確さ | △ 推測が多い | ○ 正確 | ◎ 最も正確 |
| 応答速度 | ○ 速いが手戻りが発生しやすい | △ 検索に時間がかかる | ◎ 速い＋手戻りなし |
| AI クレジット | 状況による※ | △ 多い傾向 | ○ 少ない傾向 |
| 適切な場面 | 一般的な質問 | 全体構造の把握が必要な時 | 特定ファイルの操作 |

> ※ 単発の小さなタスクではクレジット差は小さく出ることがあります。差が顕著に出るのは、複数ファイルにまたがる大きなタスクや、Agent モードで連続的に作業する場合です。

### 出力結果の観察ポイント

3 パターンの出力を見比べる際は、以下の点に注目してください：

1. **プロパティの正確さ** — `Cat` の実際のプロパティ（`Name`、`Age`、`Breed`、`Description`）を正しく使っているか？ パターン A では存在しないプロパティ（例: `Color`）を使う場合がある
2. **namespace の正確さ** — `MeowWorld.Data` を正しく参照しているか？
3. **プロジェクト規約の反映** — Custom Instructions で指定した「日本語コメント」「async/await」「LINQ メソッド構文」が反映されているか

### AI クレジットと応答品質の確認

各パターンを試した後、チャットパネル右下に表示される **モデル名** と **クレジット消費量** を確認しましょう。

![VS Code での AI クレジット消費量の確認](images/check-aicredit.png)

#### なぜクレジットが異なるのか

AI クレジット消費は以下の複合要因で決まります：

| 要因 | 影響 |
|------|------|
| **使用モデル** | Auto モードでは入力の複雑さに応じてモデルが自動選択される。明確なプロンプトは軽量モデルで処理される傾向がある |
| **入力トークン量** | `#codebase` は検索結果全体を送るため入力が多い |
| **出力トークン量** | コンテキスト不足だと推測的な前提説明が増え、出力が長くなる |
| **推論の深さ** | 曖昧な指示は「何を生成すべきか」の判断にモデルの推論リソースを使う |

> **重要な学び:** 良いプロンプトの効果は、1 回あたりの微小なクレジット差ではなく、**手戻りの削減** と **累積的な節約** に表れます。曖昧な指示で生成 → 修正を繰り返すよりも、1 回で正確な出力を得るほうが、トータルのクレジット・時間ともに大幅に節約できます。

## ベストプラクティス

### 1. 必要なファイルだけを `#file` で指定する

```
❌ プロジェクト全体のコンテキストで CRUD コントローラーを作って

✅ #file:MeowWorld/Models/Cat.cs と #file:MeowWorld/Data/AppDbContext.cs を参照して
   Cat の CRUD コントローラーを作って
```

### 2. Custom Instructions で暗黙のコンテキストを持たせる

Step 4 で設定した `.github/copilot-instructions.md` により、以下を毎回書く必要がなくなっています：

```
❌ C# で file-scoped namespace を使い、日本語コメントで、async/await を使って…

✅ Cat の CRUD コントローラーを作って
   （↑ Custom Instructions が自動適用されるので、規約の記述は不要）
```

### 3. プロンプトは簡潔に、意図を明確に

```
❌ ASP.NET Core MVC の .NET 10 プロジェクトで、Entity Framework Core を使って、
   SQLite データベースに接続して、猫の情報を管理する Web アプリケーションの
   コントローラーを作成してください。コントローラーには一覧表示、詳細表示、
   新規作成、編集、削除の機能を含めてください。

✅ Cat の CRUD コントローラーを作って。全アクション async で。
   （↑ プロジェクト情報は Custom Instructions にある。必要な指示だけ書く）
```

### 4. Agent モードの自律性を活かす

Agent モードでは、1 回の指示で複数の操作を完了させられます：

```
❌ (3回に分けて質問)
   1. Cat の CRUD コントローラーを作って
   2. 対応するビューも作って
   3. ルーティングも設定して

✅ (1回で完了)
   Cat の CRUD 機能（コントローラー + ビュー + ルーティング設定）を実装して。
   動作確認のためビルドも通して。
```

### 5. エラー発生時は `#terminalLastCommand` を活用

ターミナルでエラーが出た場合：

```
❌ (エラーメッセージをコピペ)
   以下のエラーが出ました：
   error CS0246: The type or namespace name 'AppDbContext'...

✅ #terminalLastCommand このエラーを修正して
```

## 使い分けチートシート

| やりたいこと | 推奨するコンテキスト指定 |
|-------------|------------------------|
| 特定ファイルを修正したい | `#file:対象ファイル` |
| 既存コードに合わせて新規作成 | `#file:参考にすべきファイル` を複数指定 |
| プロジェクト構造を踏まえた設計 | `#codebase` |
| エラー修正 | `#terminalLastCommand` または Agent に任せる |
| 一般的な知識を聞きたい | コンテキストなし（Ask モード推奨） |
